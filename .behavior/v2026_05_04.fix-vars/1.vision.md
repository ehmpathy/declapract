# vision: array variable support in declapract

## the outcome world

### before

practices that need lists are stuck with workarounds:

```typescript
// getVariables.ts — workaround: stringify manually
productionApprovers: '["uladkasach", "caseybrookes"]',

// resources.ts — workaround: parse it back
reviewers: { users: JSON.parse('@declapract{variable.productionApprovers}'), teams: null },
```

or worse — practices hardcode the values:

```typescript
// resources.ts — hardcoded, not reusable across orgs
reviewers: { users: ['uladkasach', 'caseybrookes'], teams: null },
```

### after

arrays just work:

```typescript
// getVariables.ts — natural array syntax
productionApprovers: ['uladkasach', 'caseybrookes'],

// resources.ts — direct substitution, no parsing
reviewers: { users: @declapract{variable.productionApprovers}, teams: null },

// output after declapract processes it
reviewers: { users: ["uladkasach", "caseybrookes"], teams: null },
```

### the "aha" moment

when a practice author writes `@declapract{variable.approvers}` and it just works — no JSON.parse, no manual stringify, no special escaping. arrays are first-class citizens like strings.

## user experience

### usecases

| usecase | goal |
|---------|------|
| github repo provisioning | declare branch protection reviewers as variable |
| ci/cd pipelines | declare allowed deploy targets as variable |
| config files | declare feature flags, allowed origins, trusted domains |
| package.json | declare workspaces, keywords, files arrays |

### contract

**step 1: user declares variables** — `declapract.use.yml`:

```yaml
declarations: npm:@org/best-practices
useCase: lambda-service
variables:
  organizationName: 'awesome-org'
  serviceName: 'svc-awesome'
  productionApprovers:
    - uladkasach
    - caseybrookes
  allowedOrigins:
    - https://example.com
    - https://api.example.com
```

yaml arrays parse naturally — no special syntax needed. the yaml parser already handles this correctly.

**step 2: practice template references variables**:

```typescript
// provision/github.repo/resources.ts (practice template)
reviewers: { users: @declapract{variable.productionApprovers}, teams: null },
origins: @declapract{variable.allowedOrigins},
```

**step 3: declapract outputs valid code**:

```typescript
// after declapract check/fix
reviewers: { users: ["uladkasach", "caseybrookes"], teams: null },
origins: ["https://example.com", "https://api.example.com"],
```

### timeline

1. user declares array variables in `declapract.use.yml` (yaml list syntax)
2. practice template references them with `@declapract{variable.X}`
3. declapract check/fix substitutes the JSON array literal
4. generated file contains valid JS/JSON array syntax

## what works vs what's broken

### already works (no changes needed)

| layer | status | why |
|-------|--------|-----|
| yaml parser | works | `js-yaml` parses yaml lists into JS arrays |
| config types | works | `ActionUsePracticesConfigInput.variables` is `Record<string, any>` |
| variable passing | works | arrays flow through to `replaceProjectVariablesInDeclaredFileContents` |

### broken (needs fix)

| layer | problem | fix |
|-------|---------|-----|
| `flatten()` call | destroys arrays — `{approvers: ['a','b']}` becomes `{'approvers.0': 'a', 'approvers.1': 'b'}` | add `{ safe: true }` option |
| replacement logic | no array handling — would output `[object Object]` | detect via `Array.isArray()`, use `JSON.stringify()` |

### the fix is localized

only one file needs changes: `replaceProjectVariablesInDeclaredFileContents.ts`

```typescript
// line 26-27: add { safe: true }
const flattenedProjectVariables: Record<string, string | string[]> =
  flatten(projectVariables, { safe: true });

// line 48: detect array and stringify
const replacement = Array.isArray(variableValue)
  ? JSON.stringify(variableValue)
  : variableValue;
return replaceAll(contents, thisVariableExpression, replacement);
```

## evaluation

### how well does it solve the goals?

| goal | solved? |
|------|---------|
| reusable practices across orgs | yes — org-specific lists come from variables |
| no workarounds | yes — no JSON.parse/stringify needed |
| backwards compatible | yes — strings still work as before |

### pros

- small code change (add `{ safe: true }` to flatten, detect array → JSON.stringify)
- type already supports arrays (`Record<string, any>`)
- natural syntax for users
- backwards compatible

### cons

- only outputs JSON array syntax (double quotes) — cannot output JS array literals with single quotes
- nested objects in arrays may produce unexpected results

### edgecases

| edgecase | behavior |
|----------|----------|
| empty array `[]` | outputs `[]` — valid |
| array with one item | outputs `["item"]` — valid |
| array of numbers | outputs `[1, 2, 3]` — valid JSON |
| array of objects | outputs `[{"a": 1}]` — valid JSON |
| nested array | outputs `[[1, 2], [3, 4]]` — valid JSON |
| null/undefined in array | outputs `[null, null]` — valid JSON |

pit of success: JSON.stringify handles all these correctly.

## open questions & assumptions

### questions (all answered)

| question | status | resolution |
|----------|--------|------------|
| does `flat` preserve arrays? | [answered] | no, must use `{ safe: true }` |
| should empty arrays error? | [answered] | no, output `[]` |
| nested objects in arrays? | [answered] | JSON.stringify handles correctly |
| mixed type arrays? | [answered] | JSON.stringify handles correctly |
| support objects too? | [answered] | deferred, out of scope |

no questions remain for [research] or [wisher].

### assumptions

1. JSON array syntax is acceptable output (double quotes, not single)
2. no need for special array access syntax (e.g., `variable.approvers[0]`)

### research completed

**`flat` library behavior with arrays:**

by default, `flat` flattens arrays into indexed keys:

```javascript
flatten({ approvers: ['alice', 'bob'] })
// → { 'approvers.0': 'alice', 'approvers.1': 'bob' }
```

to preserve arrays, must pass `{ safe: true }`:

```javascript
flatten({ approvers: ['alice', 'bob'] }, { safe: true })
// → { approvers: ['alice', 'bob'] }
```

**implication:** the implementation must pass `{ safe: true }` to `flatten()`.

### design decisions

1. **empty arrays should output `[]`, not error** — unlike empty strings (which may indicate a forgotten value), an empty array is semantically meaningful (e.g., "no approvers required"). `[]` is a valid array literal.

2. **nested objects inside arrays are valid** — JSON.stringify handles them correctly: `[{"a": 1}]` is valid output.

### no further research needed

the critical `flat` library behavior has been verified.

## what is awkward?

### double quotes vs single quotes

```typescript
// declapract outputs JSON syntax
const arr = ["alice", "bob"];

// some codebases prefer single quotes
const arr = ['alice', 'bob'];
```

this is minor — formatters/linters apply quote style. not a blocker.

### cannot output raw identifiers

```typescript
// cannot do this (would need eval)
const funcs = [functionA, functionB];

// can only output strings
const funcs = ["functionA", "functionB"];
```

this is inherent to the variable substitution model — variables are data, not code. not awkward, just a boundary.

### no partial array access

```typescript
// cannot do
const first = @declapract{variable.approvers[0]};

// must pass whole array
const all = @declapract{variable.approvers};
```

reasonable limitation — arrays are atomic values in this model.

---

**verdict**: the design is clean. JSON.stringify handles arrays correctly, the change is minimal, and the user experience is natural. no awkward tradeoffs.
